# ADR-0023: Upgrade Bootstrap from 4.3.1 to 5

## Status
Accepted

## Date
2026-03-17

## Context

Bootstrap 4.3.1 is the current version. Bootstrap 4 reached end-of-life in 2023 and no longer
receives security patches. Three concrete problems stem from remaining on BS4:

**Problem 1 — `modalService.js` is BS4-coupled and cannot be safely extended.**
`modalService.js` uses the jQuery plugin API (`$().modal('show')`, `$().modal('hide')`), which
was removed in Bootstrap 5. Any new modal feature or behaviour change requires touching the
same BS4-wired internals. ADR-0021 §4 explicitly deferred the `modalService.js` rewrite to
this milestone.

**Problem 2 — Info modal close bug (open item #1 from frontend status report).**
`closeButtonHandler()` always invokes `invokeActionAfterButtonClick('deny')`. The header ×
button also carries `data-dismiss="modal"`, so Bootstrap auto-hides the modal AND the click
handler calls `closeModal()` — producing a double-hide/remove call. The correct fix is a
dedicated close handler for info modals that only calls `closeModal()`, without invoking the
deny action. This is cleanly implementable as part of the `modalService.js` BS5 rewrite;
a BS4 band-aid would duplicate handler logic.

**Problem 3 — `bootstrap-select` is used only in two legacy views and does not support Bootstrap 5.**
`_Layout.cshtml` loads `bootstrap-select` CSS and JS. It is activated in `site.js` via
`$('.data-search').selectpicker({...})` on `<select>` elements carrying the `data-search`
class — not via the `selectpicker` class in HTML (which is why a class-name grep returns
no results). The only affected views are the legacy `Item/AddItem.cshtml` and
`Item/EditItem.cshtml` (Brand, Type, and Tags dropdowns). `bootstrap-select` does not
support Bootstrap 5. Both views are pending full rewrite as part of the legacy UI
redesign (blocked on BC atomic switches); the searchable-select UX they provide is a
candidate for re-implementation in the V2 Catalog views with a BS5-compatible library.

**Blast radius assessment.**
A workspace-wide search shows the following inline BS4 data attributes across view files:

| Location | Attribute occurrences |
|---|---|
| `_Layout.cshtml` | ~15 (`data-toggle`, `data-target`, `data-dismiss`, utility classes) |
| 11 V2 views | ~14 (collapse and modal triggers) |
| `Inventory/Index.cshtml` | 2 (inline modal trigger) |
| 19 legacy views using `modalService` | 0 — all modal HTML is generated by `modalService.js` |
| `Item/AddItem.cshtml`, `Item/EditItem.cshtml` | `data-search` class — `bootstrap-select` activated via `site.js` |

Legacy views have **zero inline BS4 modal data attributes**: they call `showConfirmationModal()`
and `showInformationModal()` via JS, which generates the modal HTML programmatically.
A `modalService.js` rewrite is sufficient to fix all 19 legacy modal views — no HTML changes
to those views are needed.

**Scope boundary.**
This ADR covers the BS5 library swap, data attribute sweep, and `modalService.js` rewrite.
It does NOT cover:
- `addObjectPropertiesToGlobal` removal (ADR-0021 tech debt — separate PR)
- Legacy view redesign (blocked on BC atomic switches — separate roadmap item)
- AMD → ESM migration (longer-term, requires full module rewiring)

## Decision

### 1. Replace Bootstrap 4.3.1 with Bootstrap 5 (target: 5.3.x)

Update `libman.json` to fetch Bootstrap 5. Remove the `bootstrap-select` library entry.
Update `_Layout.cshtml` to reference the new paths.

### 2. Sweep all inline BS4 data attributes in views

Replace BS4 data attributes with their BS5 equivalents in all affected files
(`_Layout.cshtml`, 11 V2 views, `Inventory/Index.cshtml`):

| BS4 attribute | BS5 attribute |
|---|---|
| `data-toggle="..."` | `data-bs-toggle="..."` |
| `data-target="#id"` | `data-bs-target="#id"` |
| `data-dismiss="modal"` | `data-bs-dismiss="modal"` |
| `data-backdrop="static"` | `data-bs-backdrop="static"` |
| `data-keyboard="false"` | `data-bs-keyboard="false"` |

Sweep BS5 utility class renames in the same pass:

| BS4 class | BS5 class |
|---|---|
| `ml-{n}` / `mr-{n}` | `ms-{n}` / `me-{n}` |
| `pl-{n}` / `pr-{n}` | `ps-{n}` / `pe-{n}` |
| `dropdown-menu-right` | `dropdown-menu-end` |
| `text-left` / `text-right` | `text-start` / `text-end` |
| `float-left` / `float-right` | `float-start` / `float-end` |
| `badge-{color}` | `text-bg-{color}` |
| `badge-pill` | `rounded-pill` |
| `form-inline` | flex utilities (`d-flex`, `align-items-center`, etc.) |

`input-group-append` / `input-group-prepend` wrapper `<div>`s are removed; their children
become direct siblings inside `input-group`.

### 3. Rewrite `modalService.js` using Bootstrap 5 Modal class API

Replace all jQuery plugin calls with the `bootstrap.Modal` class:

```javascript
// BS4 — removed
$('.' + modalIdentifyClass).modal('show');
$('.' + modalIdentifyClass).modal('hide');

// BS5 — replacement
const el = document.querySelector('.' + modalIdentifyClass);
new bootstrap.Modal(el, { backdrop: 'static', keyboard: false }).show();
bootstrap.Modal.getInstance(el)?.hide();
```

Replace `setAttribute('data-backdrop', ...)` / `setAttribute('data-keyboard', ...)`
with `setAttribute('data-bs-backdrop', ...)` / `setAttribute('data-bs-keyboard', ...)`.

Replace `setAttribute('data-dismiss', 'modal')` with `setAttribute('data-bs-dismiss', 'modal')`.

**Fix info modal close bug (open item #1) as part of this rewrite:**

Introduce a `closeOnlyHandler` that calls `closeModal()` without invoking
`invokeActionAfterButtonClick`:

```javascript
function closeOnlyHandler() {
    closeModal();
}

function closeButtonHandler() {
    closeModal();
    invokeActionAfterButtonClick(denyAction);
}
```

`showInformationModal` uses `closeOnlyHandler` for both the header × button and the footer
Close button. `showConfirmationModal` continues to use `closeButtonHandler` for the "Nie"
(cancel) button. Neither button carries `data-bs-dismiss` — all hiding is done exclusively
through the JS handler to avoid double-hide.

The Modal instance is stored at creation and disposed after the hide animation completes:

```javascript
let _modalInstance = null;

function showModal(el) {
    _modalInstance = new bootstrap.Modal(el, { backdrop: 'static', keyboard: false });
    _modalInstance.show();
}

function closeModal() {
    const el = document.querySelector('.' + modalIdentifyClass);
    if (!el) return;
    _modalInstance?.hide();
    el.addEventListener('hidden.bs.modal', () => el.remove(), { once: true });
    _modalInstance = null;
}
```

### 4. Replace `bootstrap-select` usage in legacy Item views

`bootstrap-select` is used only in `Item/AddItem.cshtml` and `Item/EditItem.cshtml`,
activated via `$('.data-search').selectpicker({...})` in `site.js`. The multi-select UX
(searchable tags field) is restored immediately using **Tom Select** (v2.4.1) rather than
falling back to a plain native `<select>`, because Tom Select ships a Bootstrap 5
compatible theme and is the confirmed V2 Catalog library (see §4 rationale below).

Migration steps:
- Remove the `data-search` class from all `<select>` elements in `AddItem.cshtml` and
  `EditItem.cshtml`; the elements that are NOT the tags field become plain native `<select>`.
- Remove the `.data-search` / `.selectpicker()` block from `site.js`.
- Remove all `bootstrap-select` entries from `libman.json`.
- Remove the `bootstrap-select` CSS and JS `<link>`/`<script>` tags from `_Layout.cshtml`.
- Add `tom-select@2.4.1` to `libman.json` (provider: `cdnjs`, files: `tom-select.bootstrap5.min.css` → `wwwroot/lib/tom-select/css/`, `tom-select.complete.min.js` → `wwwroot/lib/tom-select/js/`).
- Add Tom Select CSS `<link>` to `<head>` in `_Layout.cshtml` before Bootstrap CSS.
- Add Tom Select JS `<script>` to `_Layout.cshtml` after `bootstrap.bundle.min.js`.
- On the tags `<select>` in both views: add `id="ItemTagsSelect"` and `class="form-select"`.
- In each view's `@section Scripts`, inside the `DOMInitialized` handler, initialize:
  ```javascript
  new TomSelect('#ItemTagsSelect', { plugins: ['remove_button'], placeholder: 'Wybierz tagi...' });
  ```
- Run `libman restore`. Verify no 404s in browser Network tab.

Tom Select is chosen for the tags field because:
- It ships `tom-select.bootstrap5.css` which reads directly from Bootstrap 5 CSS custom
  properties (`--bs-border-color`, `--bs-primary`, `--bs-body-bg`, etc.) — no separate
  colour system, no specificity conflicts with Bootstrap styles.
- Multi-select chip removal (×) is built-in; `remove_button` plugin adds × to dropdown
  rows; `clear_button` plugin adds a clear-all button — full feature parity with
  `bootstrap-select`'s `show-tick` + `data-live-search` mode.
- UMD build available — compatible with the existing `require.js` AMD setup.
- No jQuery dependency — aligns with ADR-0021 §3 (`fetch`-first for V2 views).
- Actively maintained; latest stable release Feb 2026.

Tom Select is introduced in this migration for the `AddItem.cshtml` and `EditItem.cshtml`
tag multi-select (`#ItemTagsSelect`), loaded globally via `_Layout.cshtml`. The V2 Catalog
views should follow the same Tom Select pattern. This decision is recorded here so that
V2 Catalog view work does not re-evaluate the library choice.

### 5. Check Identity area pages

Sweep `Areas/Identity/Pages/**/*.cshtml` for inline BS4 data attributes and utility classes
and apply the same renaming rules as §2.

### 6. `frontend.instructions.md` update after merge

After this ADR is accepted and the PR is merged, update `frontend.instructions.md` to
reflect that the project targets Bootstrap 5 and that `bootstrap.Modal` class API is the
required pattern for all new modal code.

## Consequences

### Positive
- Bootstrap 4 EOL dependency eliminated — security patches resume with Bootstrap 5.x.
- Info modal close bug fixed cleanly without duplicating handler logic.
- `modalService.js` can be extended with new modal types without fighting the BS4 jQuery
  plugin API.
- `bootstrap-select` removed — ~30 kB fewer bytes loaded on every page; the two legacy
  Item views restore tag searchable-select UX via Tom Select (`#ItemTagsSelect`) loaded
  globally, and other `<select>` elements fall back to native controls pending V2 rewrite.
- V2 views gain access to BS5 utility improvements (gap, offset, logical properties).
- All 19 legacy modal views are fixed by the single `modalService.js` rewrite — no HTML
  changes to those views required.

### Negative
- `bootstrap.Modal` class API requires the `bootstrap.js` bundle to be loaded; jQuery alone
  is no longer sufficient to operate modals.
- The `closeModal()` function must now manage a `Modal` instance reference; a leaked reference
  (if `closeModal()` is not called) will prevent the next modal from opening cleanly.
- Identity area scaffolded pages may have additional BS4 references that require manual review.
- Tom Select instance on `#ItemTagsSelect` is initialised inside `DOMInitialized`; if that
  event fires before the element is in the DOM, the select will silently fall back to native.

### Risks & mitigations
- **Risk**: A view contains an inline BS4 data attribute that the search did not surface
  (e.g., inside a Razor partial or a dynamically rendered string).
  **Mitigation**: Conformance checklist item — run the grep invariant before merging.
- **Risk**: `bootstrap.Modal.getInstance()` returns `null` if the instance was not created
  via the class API (e.g., initialized by a `data-bs-toggle` attribute instead of JS).
  **Mitigation**: `closeModal()` always uses the stored `_modalInstance` reference, never
  `getInstance()`.
- **Risk**: The `hidden.bs.modal` event fires after `el.remove()` has already been called
  if the modal is removed from the DOM before the animation completes.
  **Mitigation**: The `once: true` listener removes itself; `el` is referenced via closure
  and remains valid for the duration of the hide animation even after DOM detachment.
- **Risk**: Legacy views that call `showConfirmationModal()` may pass a `denyAction`
  callback that is never invoked if the user closes the modal via the × button and
  `closeOnlyHandler` is mistakenly applied.
  **Mitigation**: `showInformationModal` uses `closeOnlyHandler`; `showConfirmationModal`
  continues to use `closeButtonHandler`. The split is enforced by the public API, not by
  a flag.

## Alternatives considered

- **Native `<dialog>` element** — rejected for now. Requires defining modal markup in HTML
  (each call site must own a `<dialog>` element) or dynamically creating one, which is
  exactly what `modalService.js` already does. Multiple simultaneous modals need multiple
  elements. BS5's `Modal` class handles stacking, backdrop, focus trap, and animation
  without custom CSS. `<dialog>` is a good long-term target once legacy views are rewritten
  and the AMD module system is replaced; it is not a prerequisite for this milestone.

- **Split-layout approach (`_LayoutV2.cshtml` for BS5, `_Layout.cshtml` stays on BS4)** —
  rejected. Legacy views have zero inline BS4 modal data attributes; the only BS4 coupling
  in legacy views is `$().modal()` calls inside `modalService.js`. Rewriting `modalService.js`
  fixes all legacy views without an HTML split. Maintaining two layouts doubles the navbar
  maintenance surface.

- **Stay on BS4, fix info modal bug with a BS4 band-aid** — rejected. The fix requires
  either duplicating handler logic or adding a flag parameter to `closeButtonHandler`. The
  deferred BS5 milestone was already planned (ADR-0021 §4). The band-aid adds work before
  the inevitable rewrite.

- **Replace `bootstrap-select` with Tom Select in the legacy Item views now** — **accepted**
  (decision revised during implementation). Tom Select v2.4.1 is installed globally and
  initialised on `#ItemTagsSelect` in `AddItem.cshtml` and `EditItem.cshtml`. The original
  rejection rationale ("unnecessary dependency surface") was outweighed by the fact that
  Tom Select is already the confirmed V2 Catalog library, ships a zero-conflict Bootstrap 5
  theme, and restores the tags searchable-multi-select UX immediately at negligible extra cost.

- **Slim Select instead of Tom Select for V2 Catalog views** — rejected. Slim Select 3.x
  ships its own CSS that does not read from Bootstrap 5 CSS custom properties; it would
  require manual overrides for every Bootstrap theme variable (colours, border radius,
  font size, focus rings). Any change to the Bootstrap theme would require re-visiting
  Slim Select overrides. Tom Select's `tom-select.bootstrap5.css` avoids this entirely.

- **Choices.js instead of Tom Select for V2 Catalog views** — rejected. Choices.js v10+
  dropped the UMD build and is ESM-only, which is incompatible with the current
  `require.js` AMD module setup without a shim. The last CJS-compatible version (v9)
  is no longer actively maintained on that branch.
